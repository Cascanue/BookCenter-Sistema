REPORTE DE ERRORES: ESPECIFICACIONES Y DIAGRAMAS VS. SISTEMA WEB

Este documento detalla todas las inconsistencias encontradas al comparar la documentación (diagramas y texto) del proyecto "BookCenter" contra el código real ejecutado en el backend (server.js).

PARTE 1: ERRORES EN LAS ESPECIFICACIONES (TEXTO DE CASOS DE USO)

Estas son las discrepancias lógicas generales que afectan tanto al texto del informe como a sus respectivos diagramas.

  • Caso de Uso 01 (Iniciar Sesión):
    El error: La especificación dice que se utiliza encriptación segura y que se registra el inicio de sesión en un log de auditoría.
    La realidad: El backend evalúa la contraseña en texto plano (ignorando bcrypt) y NO inserta ningún registro en la base de datos como auditoría (solo hace un print en consola).
  • Caso de Uso 05 (Registrar Pedido):
    El error: Indica que el sistema verifica el stock en tiempo real con la base de datos al guardar.
    La realidad: El backend (/api/guardar-pedido) guarda directamente el pedido asumiendo que está correcto. No realiza un SELECT a la tabla producto para validar si hay stock antes de insertar.
  • Caso de Uso 06 (Registrar Pago):
    El error: Indica que el sistema descuenta automáticamente el stock y que genera una boleta o factura dependiendo de si hay RUC.
    La realidad: El sistema NUNCA descuenta el stock en la base de datos. Además, la lógica para Boleta/Factura no existe; el sistema emite siempre una "Boleta" por defecto de manera forzada en el código.
  • Caso de Uso 07 (Listar/Anular Comprobante):
    El error: El flujo alterno indica que al anular, el sistema solicita un motivo, regresa el pedido al estado "Pendiente" y restaura el stock.
    La realidad: El sistema solo le cambia el estado al pedido a "Anulado". No lo devuelve a pendiente, no restaura stock y no registra motivos.
  • Caso de Uso 08 (Modificar y Eliminar Pedidos):
    El error: Describe que el Administrador puede editar cantidades de un pedido o eliminarlo por completo.
    La realidad: El servidor no tiene programadas las rutas (endpoints) para modificar (PUT) ni para eliminar (DELETE) pedidos. Solo puede listarlos.

PARTE 2: ERRORES ESPECÍFICOS EN LAS IMÁGENES (DIAGRAMAS UML)

Estos son los errores exactos ubicados en los diagramas de Actividades, Secuencia y Colaboración.

Errores Graves

  • Actividades CU-01 (image31.jpg): Muestra un fork paralelo "Registrar ingreso" en log. Esto no existe en el sistema.
  • Colaboración CU-01 (image13.png): Muestra la entidad e_logAuditoria y flecha registrarIngreso. Estos no existen.
  • Secuencia CU-01 (image28.png): Muestra la línea de vida e_logAuditoria y mensajes de auditoría que el código no hace.
  • Actividades CU-05 (image3.png): Fork stock = stock - cantidad al agregar producto. El backend NO descuenta stock al registrar un pedido.
  • Colaboración CU-05 (image8.png): Muestra un controlador c_actualizarStock y la entidad e_productos. El sistema no los utiliza en este proceso.
  • Secuencia CU-05 (image30.png): Mensajes verificarStockYPrecio() hacia e_productos. El servidor recibe todo y guarda sin consultar de nuevo a la base de datos.
  • Actividades CU-06 (image22.png): Muestra la decisión RUC → Factura/Boleta. El sistema siempre genera Boleta, sin evaluar condiciones.
  • Colaboración CU-06 (image16.png): Muestra el controlador c_calcularVuelo (calcular vuelto). El cálculo del vuelto se hace en el Frontend, no existe controlador en el servidor para eso.
  • Secuencia CU-06 (image7.png): El estado se actualiza a "Pagado". En el código real es "Completado".
  • Colaboración CU-08 (image2.png): El actor se llama Gerente. En el sistema web, el rol que existe para esto se llama Administrador.
  • Colaboración y Actividades CU-08 (image2.png): Muestran sub-flujos para "Modificar" y "Eliminar" pedido. Como se indicó antes, no existen en el código.

Errores Menores

  • Secuencia CU-07 (image15.png): Filtro por rango de fechas que no existe en el backend.
  • Secuencia CU-08 (image18.png): Filtro por estado de pedido que no existe en el backend.
  • Diagrama de Dominio (image23.jpg): Muestra Metodo_Pago como entidad independiente. En la base de datos es solo una columna de la tabla Pago.
Conclusión: Los diagramas deben simplificarse eliminando las validaciones de stock, los flujos de auditoría, las bifurcaciones lógicas de factura/boleta y los sub-flujos de edición/eliminación de pedidos para que coincidan 100% con el sistema web desplegado.
